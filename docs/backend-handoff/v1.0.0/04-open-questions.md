# v1.0.0 Open Questions

## 1. 인증

- 로그인 후 사용자 상세를 어디까지 내려줄지 결정 필요
  - 최소: `id`, `email`, `name`
  - 권장: `role`, `organization`, `permissions`
- 인증 방식 확정 필요
  - JWT access/refresh
  - server session
- 현재 프론트는 세션 만료 처리 UI가 없다. 만료 시 401 처리 정책 합의 필요

## 2. 사용자/권한

- 프로젝트가 사용자 소유인지, 조직 소유인지 결정 필요
- 사용자 역할 구분 필요
  - 관리자
  - 설계사
  - 컨설턴트
- 협업 작성자 `설계사/컨설턴트/시스템`가 표시용 문자열인지 실제 role enum인지 확정 필요

## 3. 프로젝트 모델

- `region`, `use`를 자유 입력으로 둘지 코드 테이블로 관리할지 결정 필요
- `status`를 한글 그대로 저장할지 enum code로 저장할지 결정 필요
- `map.zeb/epi/ren/consult`를 개별 컬럼으로 둘지 json으로 둘지 결정 필요

권장:

- DB에는 영문 enum code 저장
- 응답에서만 한글 label 제공

## 4. 협업 데이터 귀속

- 현재 프론트는 협업 데이터를 전역 json으로 저장하고 있어 프로젝트 분리가 없다.
- 실제 백엔드에서는 반드시 `project_id` 기준 저장 여부를 확정해야 한다.
- 추후 프로젝트 참여자별 협업 권한 제어가 필요한지 결정 필요

## 5. 첨부 파일

- 현재 프론트는 파일명/표시용 사이즈만 저장한다.
- 실제 백엔드에서는 아래를 결정해야 한다.
  - 파일 저장소 위치
  - 허용 확장자/용량
  - presigned upload 여부
  - 바이러스 검사 여부

## 6. ZEB 워크스페이스

- 시나리오를 항상 3개(A/B/C) 고정할지 확정 필요
- `savedLibrary`를 스냅샷 테이블로 영구 저장할지 결정 필요
- 분석 결과를 다시 계산 가능한 원본 파라미터만 저장할지, 결과까지 함께 저장할지 결정 필요

권장:

- 원본 파라미터와 결과 모두 저장
- 결과 재현성을 위해 계산 엔진 버전도 같이 저장

## 7. ZEB 계산 API

- step3 파라미터 명세 확정 필요
- 응답 항목 중 필수/선택 필드 구분 필요
- 계산 실패 시 에러 코드를 어떻게 내려줄지 합의 필요
- 계산 결과 캐싱 여부 결정 필요

## 8. 다국어 라우팅

- 프론트는 `src/app/[lang]` 구조를 사용하지만 일부 화면 이동은 `/login`, `/signup`, `/project-hub`처럼 locale 없는 경로를 사용하고 있다.
- 백엔드 API에는 직접 영향은 작지만, 인증 후 리다이렉트 정책과 사용자 locale 저장 여부는 정리 필요

## 9. 문서 운영 규칙

다음 변경이 발생하면 `v1.0.1` 이상 새 문서를 추가하는 것을 권장한다.

- 프론트 타입 변경
- API path 또는 request/response shape 변경
- 인증 방식 변경
- 프로젝트/협업/시나리오 저장 정책 변경
- 파일 업로드 도입
